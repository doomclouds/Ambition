# Superpowers Lite 自主子代理开发与统一审查设计

## 背景

原版 Superpowers 的子代理驱动开发默认对每个任务分别进行规格审查、质量审查和修复复审，
最后再做整体审查。该流程适合模型能力较弱、任务边界高度稳定的环境，但在当前 Codex 模型
能够理解完整仓库、主动验证并调整实现路径时，会产生大量重复评审和计划冲突：局部审查
可能强迫实现贴合已经过时的步骤，反而破坏后续任务的整体构造。

当前 `superpowers-lite` 已取消固定的逐任务双审，但 `writing-plans`、
`subagent-driven-development` 和相关模板仍保留逐交付详细验收、中途评审优先和计划步骤偏硬
的倾向。本次优化把“模型自主执行、最终统一严格审查”固化为默认路径。

## 设计决定

采用“意图契约 + 自主执行 + 高风险例外 + 最终统一审查”：

- 计划固定目标、硬约束、真实依赖、最终验收、恢复信息和已知风险。
- 任务拆分、实现方法、实验步骤和测试顺序默认是可调整建议。
- 子任务完成后只做实现者自检、相称的聚焦验证和主代理基本验收。
- 中途独立审查只用于明确的高风险例外。
- 所有任务完成后，对组合结果做一次严格的规格、质量、集成和回归审查。

技能不写死具体模型名称。自主程度由当前模型能力、上下文完整度、证据强度和任务风险共同
决定，避免规则随模型版本过时。

## 计划是意图契约

计划必须保存后续执行者不能安全猜测的内容：用户目标、兼容与权限边界、接口决定、模块
所有权、真实依赖、最终验收和恢复状态。计划不应把例行实现、每次工具调用或预想实验写成
不可偏离的施工指令。

执行期间，主代理可以根据代码现状和新证据调整任务顺序、合并或拆分任务、替换过时步骤，
以及补充实现目标所必需的工作。只要没有改变用户目标、扩大权限、突破硬约束或实质扩大
范围，就不需要暂停请示。会影响恢复或最终判断的重大偏离应简要记录原因和证据。

## 子代理执行循环

默认循环为：

1. 主代理提供目标、硬约束、依赖、禁止范围、提交权限和最终验收上下文。
2. 执行代理实现任务，自检并运行与当前风险相称的聚焦验证。
3. 执行代理报告结果、证据、计划偏离、遗留风险和下游影响。
4. 主代理检查原始证据以及下游依赖的接口或状态是否可用。
5. 没有 Critical、已知污染风险或高风险例外时，接受当前交付并继续下一项。

默认不为每个任务创建规格评审代理、质量评审代理或评审包。现有 `task-brief`、
`review-package` 和任务评审模板继续保留为按需工具，不再属于强制执行链。

## 中途高风险例外

只有以下情况默认要求暂停相关依赖或进行独立审查：

- 不可逆外部操作；
- 安全、权限或数据完整性；
- 跨模块接口、兼容性或共享状态变更；
- 当前错误会明显污染后续任务或使后续证据失真。

独立且不依赖该风险的任务可以继续。中途审查只覆盖命名风险，不扩展成完整规格与质量双审。

## 计划偏差与冲突优先级

计划和实现不一致时按三类处理：

1. 目标或硬约束冲突：立即停止相关工作并修正。
2. 实现路径变化：允许自主调整，必要时记录原因，不重新走完整规划流程。
3. 范围或授权变化：由用户决定，不以模型自主权扩大权限。

最终判断优先级为：

`用户当前意图 > 安全与兼容硬约束 > 已批准设计决策 > 真实代码契约和运行证据 > 计划建议步骤`

审查者不能仅因实现没有逐字遵循计划步骤就提出发现；必须说明偏差具体破坏了哪个目标、
契约、集成关系或风险边界。

## 最终统一审查

所有任务完成后，只对组合结果进行一次严格独立审查，覆盖：

- 用户要求和已批准设计是否完整交付；
- 跨任务接口、状态和模块所有权是否一致；
- 代码质量是否会威胁当前行为或后续安全修改；
- 集成、兼容、失败路径和回归证据是否可信；
- 遗留风险是否已解决、明确接受或正确延后。

Critical 和 Important 按依赖关系批量整理并集中修复，修复后运行一次组合复验。只有组合修复
再次改变核心语义、风险或集成置信度时，才追加一次收口复审；不恢复逐条修复、逐条复审。

## 变更范围

- `skills/writing-plans/SKILL.md`：将计划改为意图契约，降低实现步骤和实验的强制性。
- `skills/executing-plans/SKILL.md`：明确主模型的计划调整权和偏离记录边界。
- `skills/subagent-driven-development/SKILL.md`：默认连续执行，移除逐任务独立评审倾向。
- `skills/subagent-driven-development/implementer-prompt.md`：报告证据、偏离和遗留风险。
- `skills/subagent-driven-development/task-reviewer-prompt.md`：限定为高风险例外模板。
- `skills/requesting-code-review/SKILL.md` 与 `code-reviewer.md`：强化最终组合审查和偏差判断。
- `tests/contracts/skill-contracts.json`：锁定新的自主执行与统一审查语义。

如果实现检查表明某个模板已经满足设计，只增加契约或保持不动，不为了文件数量制造修改。

## 验证

1. 先修改语义契约，确认现有技能因缺少新行为而得到预期 RED。
2. 修改技能与模板后运行完整 Node 契约，要求全部 GREEN。
3. 对受影响技能运行 `quick_validate.py`，再对全部 14 个技能统一校验。
4. 运行 Codex-only 禁止路径扫描、真实 Bash 运行测试、确定性打包测试和插件校验。
5. 最后只进行一次组合审查，重点检查自主权是否清晰、硬边界是否保留、旧的逐任务双审是否
   真正退出默认路径。

## 非目标

- 不取消安全、权限、数据完整性和不可逆操作护栏。
- 不取消必要测试、适用的红绿循环或最终验证。
- 不删除任务简报、评审包和任务评审模板等按需能力。
- 不增加、删除或重命名技能。
- 不恢复 hooks、apps、MCP 或其他代理平台适配器。
- 不改变插件名称、版本、marketplace 路径和发布白名单。

## 验收标准

- 计划明确是意图与边界，不是不可调整的逐步施工脚本。
- 子代理任务默认不再触发完整规格审查、质量审查和反复复审。
- 中途独立审查只由已命名的高风险例外触发。
- 最终组合审查能够依据用户目标、硬约束和真实证据处理计划偏差。
- 契约、14 个技能、运行脚本、插件结构和确定性归档全部验证通过。
