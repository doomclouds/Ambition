# Superpowers Lite 计划执行交接设计

## 背景

当前 `brainstorming`、`writing-plans` 和 `executing-plans` 分别具备设计确认、持久计划和自主执行
能力，但阶段交接没有闭合：头脑风暴允许直接实施或写计划，计划末尾只用自然语言建议选择执行
技能，执行技能又没有把原版 todos 映射到 Codex `update_plan`。因此模型可能写完计划后直接进入
TDD 或编码，也可能根据自身判断偶尔创建任务表，行为缺少稳定契约。

`spec` 与 `plan` 连续编写、一起审查不是问题。缺口位于计划已经存在之后：书面工件没有稳定转换
成当前会话可观察、可恢复的执行状态。

## 已选方案

采用“**阶段交接硬、执行细节软**”：一旦形成书面计划并获得执行授权，必须进入一个明确的执行
工作流；执行技能按任务规模决定是否创建 `update_plan`，同时保留 GPT-5.6 对任务顺序、交付拆分、
实现方法和陈旧计划调整的自主判断。

未采用以下方案：

- 完整恢复原版：强制每个微步骤成为 todo、严格照计划施工，稳定但提示和状态维护成本过高。
- 只在 `executing-plans` 增加一句“可以使用 update_plan”：仍是可选建议，不能修复随机性。
- 为插件新增 Hook：能做运行时提醒，但超出当前 Codex-only 技能包边界，也无法替代正确的语义路由。

## 阶段交接

### 没有书面计划

已批准且只有一个短小、确定性交付时，可以直接实施。Lite 继续避免为局部机械修改制造计划仪式。

### 已形成书面计划

书面计划出现后，设计阶段的“直接实施”出口关闭。进入实现前必须选择并加载一个执行工作流：

- 用户选择当前会话执行：加载 `executing-plans`；
- 用户选择委派执行：加载 `subagent-driven-development`；
- 用户已经说“继续做”“后面不用管”“写完直接实施”等，视为提前授权继续执行，不重复询问；
- 用户把执行方式交给模型时，模型根据复杂度、隔离、上下文和并发收益选择一个工作流，并说明选择。

`writing-plans` 只负责产出和自审计划，不得在交接完成前直接修改实现文件。`using-superpowers` 的
“只加载一个主要工作流”继续适用于同一阶段，但不得被解释成跳过阶段转换；规划结束后加载执行
技能属于新的阶段。

## Codex 任务状态

`executing-plans` 开始时先判断是否需要 Codex 任务表：

- 计划包含两个或更多有意义的交付；
- 工作预计跨多个工具调用或多轮对话；
- 上下文压缩、中断恢复或用户可见进度会受益。

满足任一条件时，在修改实现文件前调用 `update_plan`。只有一个短小、可一次完成的交付时可以不
创建任务表，但仍要遵守验证、提交和恢复边界。

任务表按**有意义的语义交付**映射，不把计划文档中的每个命令、RED/GREEN 步骤或提交动作拆成
独立 todo。状态规则：

1. 创建时恰好一个 `in_progress`，其余为 `pending`；
2. 只有交付验证并提交成功后才能标记 `completed`；
3. 切换交付时先完成当前项，再让下一项进入 `in_progress`；
4. 合并、拆分、重排或替换陈旧任务时，同步更新 `update_plan` 并简述证据；
5. 不为重复命令或无语义变化制造状态更新。

`using-superpowers/references/codex-tools.md` 明确把原版 todos 映射为 Codex `update_plan`，并说明
它是会话内进度与恢复表，不替代持久计划、Git 提交或验证证据。

## 技能修改边界

- `using-superpowers`：增加阶段转换例外和“书面计划进入执行”的路由。
- `brainstorming`：明确直接实施只适用于没有形成书面计划的短小确定性变更。
- `writing-plans`：把执行交接改为必选下一技能，识别提前授权并避免重复询问。
- `executing-plans`：增加 `update_plan` 触发、粒度、状态和同步规则。
- Codex 工具参考：增加 `update_plan` 映射。
- 契约与压力场景：保护当前会话交接和多交付任务表生命周期。

## 非目标

- 不要求所有任务写 spec 或 plan。
- 不要求单交付短任务创建任务表。
- 不恢复逐微步骤 todo、固定批次大小或按计划字面执行。
- 不增加 hooks、apps、MCP、后台进程或其他代理平台适配器。
- 不把压力场景宣称为真实模型质量评测。
