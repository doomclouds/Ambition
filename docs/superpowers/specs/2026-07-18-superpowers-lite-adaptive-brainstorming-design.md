# Superpowers Lite 自适应头脑风暴设计

## 背景

当前 `brainstorming` 已经按任务风险比例适配：只处理真正开放的承重决策，不为局部确定性修改
制造设计仪式，并覆盖成功、失败、恢复、兼容和非功能边界。这个方向适合 GPT-5.6，但候选方向
探索和用户确认契约被压得过薄：技能只要求在已经存在多个实质方案时比较，没有要求模型主动
扫描候选空间；正文虽然提到用户批准，也没有明确禁止在方向仍开放时静默进入计划或实施。

原版 Superpowers `d884ae04edebef577e82ff7c4e143debd0bbec99` 强制提出 2～3 个方案、给出
推荐并等待批准，能稳定形成交互闭环；但它同时要求所有项目逐问题询问、逐节批准、写规格、
提交并转入计划，固定流程不符合 Lite 的风险比例适配目标。

## 已选方案

采用“**自适应方案漏斗 + 显式方向确认**”：恢复原版有效的方案探索、推荐和用户拍板语义，
但约束决策结果而不锁死推理过程、问题轮次、方案数量上限或文档流程。

未采用以下方案：

- 完整恢复原版流程：交互完整，但会把低风险修改也膨胀成固定设计、文档和提交仪式。
- 只增加“提供 2～3 个方案”：能改善表象，却不能阻止模型过早收敛或未经确认自行推进。
- 保持现状：自主性最高，但方向探索和批准闭环继续依赖模型临场补全，稳定性不足。

## 行为设计

### 1. 主动扫描候选空间

检查上下文后，模型应主动寻找会产生不同产品、架构或实施后果的可行方向。扫描维度由问题决定，
可以包括最小改动、长期结构、兼容迁移、失败恢复、成本、风险与可逆性；不要求逐项填表，也不为
展示数量制造弱方案。

### 2. 自适应呈现宽度

- 存在多个可行方向时，通常展示 2～4 个最值得用户选择的方案，并说明关键取舍。
- 2～4 是默认呈现宽度，不是思考上限；方向更多时可以分组或继续展开，不能丢掉会改变结果的路线。
- 只有一条路径成立时可以只推荐一个，但应说明主要替代方向因什么证据或约束被排除。
- 方案必须实质不同；禁止用明显劣化、违反约束或只是改名的方案凑数。

### 3. 推荐优先

先给出推荐方向和理由，再展示其他可行方向。推荐至少说明适用条件、主要代价、可逆性，以及它
为什么更符合当前目标。模型保持工程判断，不把所有选项摊平成无立场菜单，也不替用户拍板。

### 4. 最小必要交互

只有答案会改变候选方向、推荐或验收结果时才提问。默认围绕一个承重决策进行确认；多个问题强
相关且拆开会造成重复往返时可以成组询问。避免恢复原版“一条消息只能问一个问题”的固定轮次。

### 5. 显式方向确认门

只要仍存在会改变产品行为、总体架构、契约或实施路径的开放决定，模型必须把方案、推荐、取舍
和后果交给用户，并等待用户选择、修订或明确授权。确认前不得进入计划或实施。

用户回复方案编号、方案名称、“按推荐来”“可以”“ok”等清楚表达选择的短回复，均可作为有效
批准；方向已清楚时不得重复制造形式确认。用户否决或修订时，更新候选集或设计后再次请求确认。

### 6. 比例适配保持不变

如果代码、现有契约、事实证据或用户指令已经把局部变更收敛为唯一合理路径，它不再属于开放的
头脑风暴决策，可以说明依据后直接实施。短小确定性变更不强制设计文档；多步、跨模块或需要长期
交接的设计才固化为规格和计划。

## 契约与压力场景

静态契约应锁定以下语义：

- 主动扫描候选空间；
- 默认呈现宽度不是思考上限；
- 推荐方案及其理由；
- 方向未批准前不得进入计划或实施；
- 清楚的短回复可以作为批准，不重复询问。

新增“过早收敛且未等待批准”压力场景：用户提出仍有多条可行路线的架构选择，模型不能只给一个
结论后直接编码；它应展示实质不同的候选方案、给出推荐和证据，并等待用户选择。现有迁移完整性
和 ImageGen 可比性场景继续保留。

## 版本与发布边界

- manifest、package 和 marketplace 版本从 `0.1.4` 升至 `0.1.5`。
- 插件仍只面向 Codex，不添加 hooks、apps、MCP 或其他代理平台适配器。
- 发布根层继续只允许 `.codex-plugin/`、`assets/`、`skills/`、`README.md` 和 `LICENSE`。
- 正式归档必须从干净且已提交的实现 `HEAD` 生成。

## 验收标准

- 旧实现对新增契约或压力场景得到可解释 RED，实现后恢复 GREEN。
- `brainstorming` 能主动探索方案、推荐方向并等待用户批准，同时不恢复固定问答和文档仪式。
- 三处版本和测试期望一致为 `0.1.5`。
- Node 契约、13 个技能校验、插件校验、真实 Git Bash 运行测试、打包测试、Codex-only 扫描和
  `git diff --check` 全部通过。
- 完成归档记录交付、验证证据和正式 `0.1.5` 包摘要，提交后推送 `main` 并核对远端一致。
