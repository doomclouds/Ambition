# Superpowers Lite 高风险最小护栏设计

## 背景

当前 `plugins/superpowers-lite` 保留原版 14 个技能 ID，并把工作流改造成按任务风险与规模
使用的 Codex 中文版。该方向能减少小改动被机械展开成设计、计划、工作树、多代理和逐项
评审，但此前审查也发现三个高风险技能压缩过猛：分支收尾、评审反馈处理和系统调试仍有
少量不能依赖模型常识自行补全的操作顺序。

本次优化不恢复原版长篇流程，只补回会直接影响数据安全、仓库状态或修复质量的最小硬
护栏。

## 已选方案

采用“最小护栏 + 语义契约”方案：在三个现有技能中增加简短、可执行的强约束，并用当前
Node 技能契约锁定关键语义。

未采用以下方案：

- 回填原版对应章节：细节完整，但会重新引入固定选项、固定阶段和高提示开销。
- 只增加测试而不修改技能正文：能检测文件形状，不能改变 Codex 实际执行行为。

## 变更范围

### 分支收尾

`finishing-a-development-branch` 增加以下不可省略顺序：

1. 集成前确认相关验证可信；合并后在合并结果上重新运行与声明相称的验证。
2. 只有合并和合并后验证成功，才清理工作树或删除分支。
3. 只清理明确由当前流程创建且位于批准目录的工作树；宿主或外部工具管理的工作区不动。
4. 删除已合并分支时先移除引用它的工作树；丢弃未合并工作仍需展示影响并取得明确确认。

这些护栏不改变现有权限规则，不自动 push，也不恢复固定四选项菜单。

### 接收评审

`receiving-code-review` 增加以下依赖边界：

1. 未澄清事项会影响接口、耦合、安全或整体意图时，未澄清前不得实施依赖它的反馈。
2. 与该阻塞真正独立且明确的事项仍可继续，避免把无关修改绑成一项仪式。
3. 修复后运行聚焦验证；只有修复改变语义、风险或集成置信度时才再次独立评审。

### 系统调试

`systematic-debugging` 增加重复失败保护：

1. 每次新修复必须由新证据或可证伪的新假设支持，不能原样重试或继续叠补丁。
2. 第二次修复失败、症状跨组件迁移，或共享状态证据增强时，必须停止局部修补，重新检查
   根因、边界和架构假设。
3. 该复盘点不自动判定架构错误；只有证据支持时才升级架构处理。

## 验证

- 先在 `skill-contracts.json` 增加上述关键语义，确认现有技能契约按预期失败。
- 修改三个 `SKILL.md` 后运行完整 Node 契约，要求全部通过。
- 对三个技能运行 `quick_validate.py`，确认 frontmatter、中文内容和链接有效。
- 运行 Codex-only 术语扫描、真实 Bash 运行测试和确定性打包测试，证明没有恢复 Hook、其他
  平台适配器或污染发布归档。
- 最终只进行一次组合审查，重点判断护栏是否足够明确，同时没有重新变成重流程。

## 非目标

- 不增加、删除或重命名技能。
- 不恢复 hooks、apps、MCP 或其他平台适配器。
- 不改变插件名、版本、marketplace 路径和 rootless 发布白名单。
- 不恢复原版固定阶段、固定代理扇出、固定评审次数或固定收尾选项数量。

## 验收标准

- 三个技能都明确包含对应高风险护栏，且仍保持风险比例化和 Codex-only 语义。
- 契约能在护栏缺失时失败，并在实现后通过。
- 插件完整验证、14 个技能校验和确定性打包不回归。
