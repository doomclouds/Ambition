# Superpowers Lite 全技能深度审计与补强设计

## 背景

当前 `superpowers-lite` 已把原版 Superpowers 收敛为 13 个仅面向 Codex 的中文技能，并将
固定逐任务双审改为“模型自主执行、高风险例外、最终统一审查”。方向正确，但首次精简更重视
去除仪式，部分领域知识也被一起压薄：模型知道“要验证、要找根因、要评审”，却缺少在迁移、
持久化、并发、跨组件、测试替身和证据新鲜度等压力场景中选择正确证据的提示。

本次以原版 Superpowers `d884ae04edebef577e82ff7c4e143debd0bbec99` 为参考，不恢复原版的
强制仪式。目标是保留精简版的自主决策，把容易漏掉的领域模式放回主技能或按需参考中。

## 设计原则

1. **短主干、深参考：** 主 `SKILL.md` 保存触发条件、决策门禁和常见失败；特殊领域模式进入
   一层相对链接，避免每次加载整本教材。
2. **规范约束结果，不锁死施工顺序：** 测试、调试和评审必须取得能区分真假的证据，但模型可
   根据仓库和风险选择测试层级、实验、任务顺序和工具。
3. **最终统一审查：** 不恢复每个子任务的完整规格审查和质量审查。中途只处理会污染后续构造的
   已命名高风险，所有任务完成后统一检查组合结果。
4. **压力场景优先：** 静态短语契约之外，增加覆盖 13 个技能的场景目录，锁定压力下必须做出的
   决定、禁止捷径和所需证据。
5. **不按行数评判：** 已经完整的技能保持不动；只有缺失会导致真实误判时才扩写。
6. **任务提交是恢复边界：** 每个计划任务或可独立验收交付在验证通过后必须形成独立 Git
   提交，未提交不得标记完成或进入下一任务。计划开始前一次性确认提交权限；没有权限或提交
   失败属于阻塞，不能把多项差异长期堆在工作树中。

## 结构设计

### 主技能补强

- `brainstorming` 增加成功、失败、恢复、兼容与非功能边界的设计检查，但仍只在真正开放的承重
  决策上请求批准。
- `dispatching-parallel-agents` 增加独立性矩阵、最小上下文包、外部状态所有权、部分失败处理和
  主代理组合复验。
- `executing-plans` 增加压缩恢复、陈旧计划修正、失败证据分类和可逆回退。
- `finishing-a-development-branch` 增加仓库/工作树/HEAD/远端状态检测、脏差异保护、集成后验证和
  清理所有权。
- `receiving-code-review` 增加反馈来源校准、相关项成组、实际使用搜索、冲突优先级和限制报告。
- `systematic-debugging` 增加复现质量、多组件边界取证、正常样本对比、单一假设实验和间歇性故障。
- `test-driven-development` 明确有效 RED 的判据、测试层级选择、状态与交互风险、既有实现的反事实
  证据和测试替身边界。
- `verification-before-completion` 建立“声明—证据”门禁，要求完整命令、退出码、失败数、语义回读
  和证据限制。
- `writing-plans`、`executing-plans` 与 `subagent-driven-development` 统一规定：计划开始前确认提交
  权限，每个任务验证通过后只提交该任务差异，提交成功才更新为完成并继续；不把最终统一审查
  误解为最终才统一提交。

### 按需参考

- 新增 `test-driven-development/test-selection-patterns.md`，覆盖纯逻辑、状态持久化、API/CLI、
  并发时间、UI、迁移兼容、外部依赖和非代码交付的测试选择。
- 新增 `verification-before-completion/evidence-matrix.md`，按完成声明给出最强证据与常见伪替代。
- 扩展 `requesting-code-review/code-reviewer.md`，覆盖数据迁移与回滚、并发共享状态、错误恢复、
  权限安全、兼容、真实行为测试、可观测性和证据限制。
- 扩展 `systematic-debugging/root-cause-tracing.md` 与 `defense-in-depth.md`，保留一层引用。

### 压力场景契约

新增 `tests/evals/workflow-pressure-scenarios.json`。每个场景包含技能、压力条件、期望决定、禁止
决定和所需证据。契约要求 13 个技能全覆盖，且对最容易误判的 TDD、调试、验证和最终评审提供
多个场景。场景目录是可审计的行为要求，不伪装成真实模型评测分数。

## 版本与发布边界

- manifest、package 和 marketplace 条目从 `0.1.2` 升至 `0.1.3`。
- 仍只发布 `.codex-plugin/`、`assets/`、`skills/`、`README.md` 和 `LICENSE`。
- 不恢复 `writing-skills`，不添加 hooks、apps、MCP 或其他代理平台适配器。
- 用户已授权并要求每个完成交付形成独立 Git 提交；未授权推送。正式 `0.1.3` 归档只能在全部
  交付提交、统一验证通过且 `HEAD` 干净后生成。

## 验收标准

- 13 个技能都有明确审计结论，压力场景契约完整覆盖。
- 被判定过薄的技能能处理其命名压力场景，同时不恢复固定逐任务评审和强制施工顺序。
- 新参考只从对应主技能一层链接，公开 Markdown 保持中文，代码注释保持英文。
- 插件契约、技能契约、13 个 `quick_validate.py`、插件校验、运行脚本、打包行为和
  `git diff --check` 全部通过。
- 最终统一审查没有未关闭的 Critical 或 Important；未运行的正式发布步骤及原因明确记录。
- 规格、RED 契约、工作流补强、质量技能补强和最终验证分别形成可恢复提交；任何计划任务都不以
  未提交差异状态宣告完成。
