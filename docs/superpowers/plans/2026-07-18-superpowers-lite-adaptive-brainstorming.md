# Superpowers Lite 自适应头脑风暴实施计划

**目标：** 让 `brainstorming` 在保持 GPT-5.6 自主判断和风险比例适配的同时，稳定主动展开候选
方向、给出工程推荐，并在开放的承重决策上等待用户明确确认；发布 `0.1.5`。

**稳定引用：** `docs/superpowers/specs/2026-07-18-superpowers-lite-adaptive-brainstorming-design.md`、
根 `AGENTS.md`、原版 Superpowers `d884ae04edebef577e82ff7c4e143debd0bbec99`、当前
`brainstorming`、插件契约和压力场景。

**仓库操作：** 用户已授权直接在当前干净 `main` 修改，要求版本号加一、完成后提交并推送远端。
每个独立交付验证后单独提交；不创建 worktree，不创建 PR。推送后核对本地与 `origin/main` 一致。

**恢复状态：** 基线为 `cb0f60b`、版本 `0.1.4`、工作树干净且与 `origin/main` 一致。恢复时先读
本计划、设计规格、`git status` 和最近提交；只有当前命令输出支持完成状态。设计与计划提交后，
先建立 RED 契约，再实现正文和版本，最后从干净实现提交生成正式包并归档。

## 任务 1：固化设计与执行边界

**交付结果：** 已批准的自适应方案漏斗、显式确认门、非目标、版本和发布边界可以脱离当前对话
独立恢复。

**硬约束与依赖：** 不恢复原版固定问答轮次、固定方案上限、逐节批准或所有变更强制文档；版本
精确升级到 `0.1.5`；用户已授权提交和推送。

**候选路径：** 新增本设计规格和实施计划，回读并运行 `git diff --check`。

**连续性证据：** 两份文档完整覆盖用户要求、候选方案、确认语义、版本和验证边界。

**恢复检查点：** 提交成功后，下一任务只修改契约、压力场景和版本期望以取得 RED。

**提交边界：** 只提交本规格和计划；提交成功才进入 RED。

## 任务 2：定义自适应探索与确认契约

**交付结果：** 自动化契约能识别过早收敛、缺少推荐、静默进入实施和版本未升级。

**硬约束与依赖：** 压力场景是可审计的行为要求，不伪装成真实模型质量分数；现有场景不得删除；
版本期望精确为 `0.1.5`。

**候选路径：** 扩展 `tests/contracts/skill-contracts.json`，新增
`brainstorming-premature-convergence-without-approval` 压力场景及必需 ID，更新三处插件版本测试
期望。旧正文和元数据上运行 `npm test`，取得由目标行为缺失导致的 RED。

**连续性证据：** 失败应命中新正文语义或旧版本，不得由 JSON、路径或测试基础设施错误造成。

**恢复检查点：** 记录通过/失败数量和目标失败项；提交后下一任务实现对应正文与版本。

**提交边界：** 只提交契约、场景和版本测试期望；RED 证据可写入提交说明，提交成功才实现。

## 任务 3：实现头脑风暴闭环并发布 0.1.5 元数据

**交付结果：** `brainstorming` 主技能主动扫描候选空间，通常展示 2～4 个实质方案、推荐优先、
等待批准并识别清楚的短回复，同时保留确定性局部变更直接实施的 Lite 边界。

**硬约束与依赖：** 2～4 是默认呈现宽度而非思考上限；不得凑弱方案；真正开放的承重决定批准
前不得进入计划或实施；三处版本同步为 `0.1.5`。

**候选路径：** 重写 `brainstorming/SKILL.md` 的决策落地与批准段落，必要时调整标题；更新
`.codex-plugin/plugin.json`、`package.json` 和 marketplace。运行 Node 契约、技能校验和
`git diff --check`。

**连续性证据：** `npm --prefix .\plugins\superpowers-lite test` 全部 GREEN，`brainstorming` 的
`quick_validate.py` 输出 `Skill is valid!`。

**恢复检查点：** 实现提交后工作树必须干净；正式归档只从该提交生成。

**提交边界：** 只提交技能正文和三处版本元数据；聚焦验证通过后独立提交。

## 任务 4：完整验证、归档与远端交付

**交付结果：** `0.1.5` 具备完整本地验证、正式包摘要、可检索归档和远端可用提交。

**硬约束与依赖：** 正式包从干净实现 `HEAD` 生成；归档根层白名单不变；Git Bash 使用明确路径；
推送前完成资产 gate 和归档，推送后核对 ref 相等。

**候选路径：** 运行 Node 契约、13 个 `quick_validate.py`、插件校验、两个真实 Git Bash 测试、
Codex-only 扫描和 `git diff --check`；从实现提交生成 `superpowers-lite-0.1.5.zip` 并记录条目数与
SHA-256；更新验证记录、新增 archive 和索引，运行完成 gate，提交文档后复跑聚焦验证并推送。

**连续性证据：** 完整命令、退出码、通过数量、归档条目和哈希写入验证记录与 archive。

**恢复检查点：** 文档提交后确认工作树干净；推送失败时保留提交并报告精确错误，不改写历史。

**提交边界：** 只提交验证记录、archive 和索引；提交成功后推送 `main`，核对
`git rev-parse HEAD` 与 `git rev-parse origin/main` 相等。
