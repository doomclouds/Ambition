# ImageGen 视觉伴侣

使用 Codex/ChatGPT **内置 ImageGen** 把真正依赖视觉判断的开放决策变成高保真图片，并在当前
对话中**直接内联展示**。图片帮助用户看清差异，文字负责记录准确的决定。

## 判断是否需要图片

按当前问题判断，不要把整场头脑风暴都变成出图流程。

适合生成图片：

- UI 页面、组件层级、导航和布局密度；
- 视觉风格、配色、材质、光影和品牌气质；
- 产品外形、空间关系、场景构图和交互状态；
- 需要并排观察的 2～4 个设计方向；
- 现有图片上的聚焦视觉修改。

继续使用文字、表格、Mermaid 或代码原生表达：

- 范围、需求定义、优缺点和非视觉取舍；
- 接口、数据模型、精确流程、时序和状态契约；
- 必须逐字准确的长文案、数字、命令和验收条件；
- 图片不能提供额外判断依据的问题。

涉及 UI 不代表必须生成图片。“是否需要管理员角色”是产品决定；“管理员控制台采用哪种布局”
才是视觉决定。

## 唯一能力路径

加载当前 Codex 的 `imagegen` 技能，使用内置 `image_gen` 工具完成生成和编辑。不要自行搭建本地
视觉服务，也不要改走其他生成路径。

如果内置能力不可用或**生成失败**：

1. 明确说明本轮没有成功生成视觉稿；
2. 保留已经确认的文字结论和开放问题；
3. 用文字继续推进不依赖视觉的决定，等待能力恢复后再处理视觉问题；
4. 不把未生成、未展示或未检查的图片描述成已完成结果。

## 生成前组织问题

先用一句话说明图片要帮助决定什么，再整理最小生成规格。只加入能改善判断的细节：

```text
Use case: ui-mockup | product-mockup | stylized-concept | infographic-diagram
Asset type: 图片的用途
Primary request: 当前要比较或验证的视觉问题
Subject: 主体、真实内容和必要状态
Style/medium: 目标保真度与视觉语言
Composition/framing: 画布、视口、构图和信息层级
Text (verbatim): 必须出现的短文字
Constraints: 必须保持的不变量
Avoid: 不得出现或不得改变的内容
```

用户已经给出具体要求时，只做结构化整理，不擅自增加品牌、角色、口号或故事元素。需求较泛时，
可以补充必要的构图、保真度和实际使用场景，但不能替用户发明新的产品决定。

## 生成可比较的方案

需要比较方向时，优先制作一张带 A/B/C 标识的对比图；细节在同一张图里过小时，再分别生成，
但所有方案必须保持：

- **相同的画布**、设备视口和观察角度；
- 相同的真实内容、数据量、功能范围和状态；
- 相同的保真度、品牌基线和输出用途；
- 只改变当前要决定的主要变量。

不要在一轮里同时改变信息架构、配色、字体、内容密度和交互方式，否则结果看起来不同，却无法
知道用户真正选择了什么。每轮最多比较 2～4 个方向。

生成后在图片旁用简短文字列出：

- A/B/C 各自只改变了什么；
- 每个方向的主要收益和代价；
- 图片里可能不准确但决定中必须准确的文字或数字；
- 希望用户回复的选择，例如“选 A，保留 B 的筛选区”。

## 精确文字与技术事实

图片不是规格数据库。即使提示中提供了 **精确文字**，也要检查成图，并在对话中逐字重述关键
文案、数字、接口名和约束。用户的决定以对话和最终设计文档为准，不以位图中的字符为准。

如果图片中文字错误：

- 先判断文字是否真的需要出现在图中；
- 必须出现时，缩短数量，逐字引用，并要求不得增加其他文字；
- 仍不稳定时让图片只表达结构与风格，把准确内容放在图片旁。

精确架构、流程或数据关系优先使用 Mermaid、表格和代码原生图示。ImageGen 可以表达概念气质，
不能代替可验证的技术契约。

## 参考图与编辑

使用已有图片时，先标明每张图片的角色：

- 编辑目标：需要保留大部分内容，只修改指定部分；
- 风格参考：借用风格、构图或氛围，不直接修改原图；
- 内容参考：提供产品、人物或场景事实。

编辑时重复不变量，例如“只改变侧栏宽度；保持内容、配色、视口和其他布局不变”。检查输出的
主体、构图、文字和禁止项后再展示。

收到反馈后，优先使用上一张图片作为编辑目标或参考，**一次只改变一个主要变量**。不要每轮
重写整套提示，避免已经批准的方向漂移。

## 展示与反馈

每次生成后：

1. 检查主体、构图、选项可比性、文字和不变量；
2. 直接内联展示图片，不只报告文件位置；
3. 说明用户正在看什么以及本轮要决定什么；
4. 请用户用自然语言或 A/B/C 回复，图片本身不承担交互状态；
5. 把选择、理由和保留项写回设计结论，再进入下一个承重问题。

视觉伴侣是决策证据，不是批准本身。用户的文字反馈和明确批准仍是最终输入。

## 预览与项目资产

- 只用于探索或头脑风暴的图片属于预览：内联展示即可，可以保留在内置能力的默认生成位置。
- 被用户批准且将由代码、文档或设计规格引用的图片属于**项目资产**：完成前复制到工作区的稳定
  路径，遵守仓库既有命名和资产目录约定，并报告绝对路径。
- 不覆盖既有资产，除非用户明确要求替换；默认使用语义化版本名，例如
  `dashboard-options-v2.png`。
- 项目引用不得只指向 Codex 的默认生成目录，否则换机器或清理缓存后会失效。

## 完成检查

进入下一项决定前确认：

- 图片确实回答了一个视觉问题；
- 多方案使用了统一输入，只改变了明确变量；
- 关键精确内容已经在对话中重述；
- 用户选择、修改意见或未决点已记录；
- 需要长期使用的最终图片已保存为项目资产；
- 没有把生成失败、未检查输出或预览图误报为完成交付。
