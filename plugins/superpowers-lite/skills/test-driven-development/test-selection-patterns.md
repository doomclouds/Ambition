# 测试选型参考

## 目的

测试层级不是框架偏好，而是风险位置的函数。先定位行为在哪个**状态、边界或交互**上可能出错，
再选择最小但足以区分真假的证据。快速测试负责定位，贴近生产的测试负责证明不能被低层替代的
语义；两者按风险组合。

## 快速决策表

| 行为风险 | 首选证据 | 常见补充 | 不足替代 |
| --- | --- | --- | --- |
| 纯计算与不变量 | 单元测试、性质测试 | 边界值、变异 | 只测一个示例 |
| 状态与持久化 | 真实存储或可信临时实例 | 重启、并发、失败恢复 | 只检查内存对象 |
| API/CLI/协议 | 契约或进程级测试 | 错误路径、消费者测试 | 只调用内部函数 |
| 并发与时间 | 条件控制、确定性调度 | 压力测试、真实时钟抽样 | 固定 sleep |
| UI/交互 | 组件交互或端到端操作 | 视觉、无障碍、设备检查 | 只测 view-model |
| 迁移与兼容 | 旧数据/旧消费者跨版本 | 中断、回滚、重复执行 | 只测全新数据库 |
| 外部依赖 | 合同测试 + 最小测试替身 | 沙箱或真实探针 | 复杂 mock 自测 |
| 配置/文档/生成物 | 解析、生成、渲染和回读 | 安装/运行路径 | 文本包含断言 |

## 纯逻辑与不变量

适用于解析、转换、计算、校验和状态机中的局部规则。优先测试公开行为和不变量，而非私有实现
步骤。边界值来自领域：空、最小/最大、重复、顺序、精度、非法组合和溢出。输入空间大且规则
稳定时，可用性质测试证明恒真条件；不要为了覆盖率给每个分支复制相似样例。

有效 RED 应因规则尚未存在或错误而失败。若测试在实现前就通过，检查它是否没有进入目标分支，
或断言只验证了输入原样返回。

## 状态、持久化与重启

持久化行为至少区分内存状态、写入确认和重新读取：

1. 从已知初始状态执行真实写入；
2. 释放对象、连接或进程，避免命中内存缓存；
3. 重启或重新加载后回读；
4. 检查版本、顺序、幂等性和不应改变的字段；
5. 在写入中间制造失败，确认事务、补偿或恢复语义。

文件测试使用独立临时目录并验证真实字节、权限或原子替换；数据库测试优先使用与生产语义相符
的临时实例或容器。内存替身只有在它能保留事务、唯一性、排序和并发契约时才足够。

## API、CLI 与边界契约

内部函数通过不能证明真实入口可用。对 HTTP、RPC、CLI、插件或序列化边界验证：

- 参数解析、默认值和配置优先级；
- 真实编码、字段命名、退出码、状态码和错误正文；
- 身份、权限和敏感信息处理；
- 超时、取消、重试和部分成功；
- 副作用发生一次且最终状态正确；
- 至少一个代表性消费者能使用产物。

协议契约稳定时，生产者和消费者可共享模式或契约用例，但不要共享同一错误实现作为“真相”。

## 并发、时间与随机性

先把不可控条件变成可观察条件：门闩、事件、屏障、假时钟、可注入随机源或确定性调度器。假时钟
必须保持业务对时间单调性、时区和截止期限的语义，不能只让测试跑得快。

并发测试关注不变量和最终状态，而不固定线程的偶然顺序。覆盖重复投递、取消与完成竞态、锁超时、
幂等键、共享缓存失效和资源释放。压力测试可以提高发现概率，但它不能替代一个能解释失败条件的
确定性用例。

## UI、视觉与真实交互

把可独立验证的状态转换留在低层；对焦点、键盘、拖放、动画、布局、响应式、无障碍树和平台
集成，使用真实渲染与操作。视觉快照适合捕获稳定布局回归，不适合替代“按钮是否真的完成操作”。

复杂交互至少验证初始状态、用户动作、加载/错误/空状态、最终状态和刷新后的保持。涉及系统对话框、
剪贴板、文件选择或浏览器权限时，补充真实环境或受控端到端证据，并明确自动化无法触达的部分。

## 迁移、兼容与回滚

迁移测试从真实旧版本数据或工件开始，而不是只创建最新模式：

- 旧数据升级后可读、语义不丢失；
- 迁移重复执行安全，或明确拒绝；
- 中途中断后能恢复、继续或回滚；
- 滚动升级期间新旧版本的读写组合符合承诺；
- 降级或回滚不会让已写入数据静默损坏；
- 大数据量下的锁、时间和空间成本可接受。

兼容性声明需要代表性旧消费者、旧配置和旧产物。类型检查通过不等于二进制、协议或行为兼容。

## 外部依赖与测试替身

使用**测试替身**前先列出真实依赖的契约：输入、输出、错误、时序、副作用、重试、幂等性和资源
生命周期。选择替身类型：

- Stub 提供少量固定响应；
- Fake 保留关键语义的轻量实现；
- Spy 观察必要交互，但不成为测试目标；
- Mock 约束交互，只在交互本身就是契约时使用；
- 沙箱或真实服务探针验证替身没有漂移。

替身不应跳过被测行为依赖的副作用。若必须维护大量响应字段、调用顺序和内部细节，说明边界可能
选错；优先下移替身到真正外部的层，或上移为集成测试。

## 配置、生成物与文档

没有传统运行时代码不等于不可验证：

- 配置：用真实解析器读取，检查默认值、覆盖优先级和运行时回读；
- 生成物：运行生成器，检查结构、确定性、可执行性和消费者兼容；
- 安装/打包：从干净输入生成，检查白名单、元数据、权限和可重复性；
- 文档：执行命令、渲染页面或按新用户路径走通，检查链接和示例；
- 提示词/技能：静态契约锁定硬边界，压力场景说明预期决定，必要时再做真实模型评测。

文本包含测试只能证明词存在，不能证明行为完整。它适合防止关键规则被意外删掉，必须与人工语义
回读、场景契约或运行证据配合。

## 组合证据

不要追求“最高层级都测一遍”。选择组合时问：

1. 哪个快速测试能精确定位逻辑回归？
2. 哪个边界测试能证明真实协议或状态生命周期？
3. 哪个语义回读最接近用户或运维结果？
4. 哪些风险仍无法自动化，如何明确报告？

一个可靠组合通常是少量聚焦测试加一条真实边界验收，而不是大量 mock 单测或脆弱的全端到端套件。
