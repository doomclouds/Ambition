# 测试驱动开发

## 目的

TDD 用来取得能区分真假的证据：测试必须证明自己能检测缺失或损坏的行为，而不是只在当前实现上通过。当行为变化存在有效自动化测试缝隙时，红绿循环是首选基础；但它不是删除有用现有代码的理由，也不能把字面执行顺序误当成正确性。

## 选择验证路径

1. 明确变更必须产生的真实产品、业务或运维语义，包括有意义的失败与边界情况。
2. 写出**测试选择依据**：风险主要位于计算结果、状态生命周期、模块边界还是外部交互？选择能在
   最低合理层级观察该风险的测试，而不是默认只写单元测试。
3. 如果存在有效自动化测试缝隙，编写或收紧一个聚焦测试，取得反事实证据，证明测试确实因预期缺失或损坏的行为而失败。随后实施或修复行为，并验证测试通过。
4. 如果实现已经存在，保留它，同时构建回归测试与反事实证据。可以使用修复前复现、受控夹具、聚焦变异，或其他能诚实证明断言可区分缺陷的方法；不要只为重建操作顺序而删除正常代码。
5. 对配置、生成物、文档，或没有有用自动化缝隙的既有修复，选择最强适用验证：解析并回读配置、检查生成物、渲染并实际走通文档工作流，或运行窄而贴近生产的检查。

当变更跨越状态、边界或交互，或不确定单元、集成、契约、端到端和回读如何组合时，阅读
[测试选型参考](test-selection-patterns.md)。

## 适用时执行红绿循环

对于有意义的自动化测试缝隙：

- **RED：** 陈述预期行为，运行聚焦测试，并检查失败原因。有效 RED 必须满足：测试成功启动，
  断言或可观察结果与目标行为相关，失败信息符合预期，并且**失败必须能区分目标行为**缺失或损坏。
  语法错误、导入错误、环境未启动、夹具损坏和错误断言不是预期缺陷的证据。
- **GREEN：** 实现满足场景的最小行为变化，然后重跑聚焦测试。
- **REFACTOR：** 只有在保持测试区分行为能力的前提下，才改进代码或测试。针对不同的真实风险增加用例，不要凑数量。

测试在修改前就通过时，先判断行为是否已经存在、断言是否观察错位置、场景是否没有触发风险，
或测试是否只验证了替身。不要继续写实现再把既有绿色包装成 RED。回归测试面对已经修复的代码时，
使用修复前提交、受控故障注入、聚焦变异或真实历史输入取得诚实反事实。

单元测试、编译、lint 和类型检查只是支持信号。必须配上与任务相符的语义行为证据：前端工作要渲染并检查交互，持久化要重启并回读，API 或自动化要检查实际参数与最终状态，兼容性工作要验证代表性消费者产物。

## 选择测试层级

- 纯函数和局部不变量优先使用快速、聚焦的单元测试；
- 状态、持久化、缓存和生命周期要验证保存、重载、重启、失效和恢复；
- API、CLI、序列化和插件边界要验证真实协议、错误映射和代表性消费者；
- 并发、时间、重试和随机性要控制条件，不能靠扩大 sleep 或重复到通过；
- UI 要区分可单测的状态逻辑与必须渲染、操作、截图或无障碍检查的真实交互；
- 迁移和兼容要跨旧数据/旧消费者运行，并覆盖失败中断与回滚。

**测试替身**只隔离真正缓慢、非确定或不可控的边界。替身必须保留测试依赖的契约和副作用，
且断言目标仍是被测行为。Mock 设置比行为更复杂时，优先考虑真实组件、轻量假实现或更高层测试。

## 证据记录

记录场景、可用的反事实或 RED 证据、GREEN 结果，以及支持完成声明的语义回读。如果测试本身不能证明用户相关行为，明确指出哪项回读补上了缺口。

## 红旗

- 测试通过，但没人知道它是否真的能捕获缺陷。
- 全绿单元测试替代了从未执行的用户可见、持久化、运维或兼容性结果。
- 只为满足程序化顺序而删除或重写有用实现。
- 明明存在更强的验证路径，却把配置、生成物、文档或既有修复当成“不可测试”。
- RED 实际是测试错误、环境错误或错误夹具，却继续进入 GREEN。
- 测试只验证调用次数或 mock 返回值，没有观察真实契约和最终状态。
- 测试层级方便但不触达风险，例如持久化修改只测内存对象。

编写或修改测试、增加 mock，或想在生产代码中添加测试专用方法时，阅读[测试反模式](testing-anti-patterns.md)。
