# 声明—证据矩阵

## 使用方法

先写出准备对用户说的具体声明，再从下表选择能直接支持它的最强证据。右栏列出常见但不足的
替代物。证据可以组合，但多个弱信号不会自动等于一个缺失的语义验收。

## 声明—证据矩阵

| 声明 | 最强适用证据 | 常见不足替代 |
| --- | --- | --- |
| 聚焦测试通过 | 运行目标测试的**完整命令**，退出码 0、0 失败并读完输出 | 上次输出、IDE 绿色图标、应当通过 |
| 完整测试通过 | 当前 HEAD 上运行项目批准套件，记录数量、失败、跳过和退出码 | 只跑受影响文件、lint 通过 |
| 构建成功 | 真实发布/目标配置构建退出 0，产物存在且可读取 | 类型检查、单元测试、旧产物 |
| 缺陷已修复 | 重现**原始症状**的场景先能区分失败，修复后通过，并检查邻近回归 | 代码 diff、没有再次看到报错 |
| 回归测试有效 | RED 因目标缺陷失败、GREEN 因修复通过；既有修复用诚实反事实 | 新测试只运行一次且通过 |
| 功能完成 | 要求映射 + 自动化证据 + 用户/运维语义回读 | 测试全绿但遗漏要求 |
| API/CLI 可用 | 真实入口、有效与无效参数、退出/状态码、错误正文和最终副作用 | 直接调用内部函数 |
| 持久化可靠 | 写入、关闭/重启、回读；失败中断与恢复后状态符合契约 | 内存对象值正确 |
| 迁移可用 | 旧数据升级、重复/中断、回滚或恢复、滚动版本组合 | 新建最新空数据库通过 |
| 向后兼容 | **代表性消费者**、旧配置或旧产物跨承诺边界运行 | 编译当前项目、类型相同 |
| UI/交互正确 | 实际渲染和关键操作；按风险补视觉、无障碍和设备证据 | view-model 单测、静态 HTML |
| 性能改善 | 同环境、同数据、预热与多次测量，报告分布和资源代价 | 单次更快、日志时间戳 |
| 安全边界成立 | 授权矩阵、负向用例、真实身份/权限和副作用回读 | 只检查输入验证函数 |
| 打包/发布可用 | 干净已提交 HEAD 生成包，白名单、元数据、权限、安装/消费者和确定性 | 源码目录看起来完整 |
| 合并可用 | 精确合并结果上重跑相称验证，确认冲突解决和目标分支状态 | 功能分支合并前通过 |
| 代理任务完成 | 检查提交/diff、运行原始验证、核对下游接口和最终状态 | **代理报告** DONE |

## 证据新鲜度

证据是否陈旧取决于相关状态变化，而不只是时间：

- 源码、测试、依赖、配置、数据、环境、生成器或平台变化可能使对应证据失效；
- 合并和冲突解决产生新代码组合，通常需要在合并结果上复验；
- 只修改不影响运行的说明文档，不必机械重跑昂贵性能套件；
- 无法确认旧证据是否仍适用时，先运行聚焦探针，再决定复用或扩大验证。

每条复用证据都应能回答：它针对哪个提交或工作树状态，之后发生了什么，以及这些变化为什么
不会改变结论。

## 读取完整结果

命令退出 0 仍要检查：

- 测试是否实际发现并运行，还是 0 tests；
- 是否存在跳过、隔离、重试后通过或允许失败；
- 构建是否使用了缓存或旧工件；
- 脚本是否把错误写到日志却吞掉退出码；
- 产物路径、版本和消费者是否是目标对象；
- 警告是否直接触及当前声明，例如弃用 API、签名失败或迁移跳过。

退出非零时报告实际失败，不从部分通过的子项外推整体成功。

## 语义验收与支持信号

编译、lint、类型检查、单元测试和静态短语契约是重要支持信号，但它们各自证明的范围有限。
将声明拆成层次：

1. **结构正确：** 格式、类型、链接、清单和静态约束；
2. **局部行为：** 聚焦自动化测试能区分目标逻辑；
3. **边界行为：** 真实协议、进程、存储、生成物或平台交互；
4. **用户/运维结果：** 完整场景的最终状态、恢复和体验。

声明达到哪一层，证据至少要触达哪一层。无法触达时降低声明强度并说明**证据限制**。

## 委派与评审结果

代理、评审者和 CI 摘要可以帮助定位证据，但不是自动可信的完成证明。至少检查：

- 它评估的精确基线、HEAD 或工作树是否正确；
- 报告中的命令是否真的运行，输出与退出码是否可回读；
- 提交是否只包含任务范围，是否遗漏未提交差异；
- 依赖任务看到的接口和状态是否与报告一致；
- 多个独立绿色结果组合后是否仍满足共享状态和消费者契约。

不能访问原始证据时，明确说“目前只有代理报告，尚未独立验证”，不要换一种措辞宣布完成。
