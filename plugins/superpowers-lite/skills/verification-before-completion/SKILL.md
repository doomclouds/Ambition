# 完成前验证

## 目的

与用户相关的正确性和完成声明必须有当前可信证据。验证的单位不是“我运行过一些命令”，而是
一个具体**声明—证据**关系：这份证据能否证明这句话，并覆盖用户真正关心的语义。

## 声明—证据门禁

做出完成、修复、通过、可合并或可发布声明前：

1. 精确写出准备声明什么，范围到模块、平台、数据或消费者；
2. 选择能直接证明它的最强可用证据，而不是最方便的支持信号；
3. 运行**完整命令**或执行完整语义场景，读取全部相关输出；
4. 检查**退出码和失败数**、跳过项、警告、生成物以及最终状态；
5. 判断输出是否真的支持声明；不支持就报告实际状态，不用“应该”“看起来”补齐；
6. 记录任何**证据限制**：未覆盖的平台、消费者、数据规模、权限、外部服务或人工步骤。

完整命令是能证明当前声明的完整验证单位，不等于每次都跑仓库最大套件。聚焦修复可以先用聚焦
测试；声明整个插件可发布时则需要完整契约、结构、脚本、打包和消费者边界。常见声明的最强证据
和伪替代见[声明—证据矩阵](evidence-matrix.md)。

## 判断证据新鲜度

依赖已有证据前，先判断证据产生后哪些状态发生过变化：源代码、依赖、配置、生成物、环境、数据、部署边界或场景本身。只有相关变化可能使证据失去原意时，它才是陈旧证据；不能只因时间过去就判它过期。

新鲜度不确定时，运行一个聚焦探针检查有争议的状态。根据结果决定旧证据是否仍可信，还是必须重新生成更广的测试、构建、回读或贴近生产的场景。验证广度必须匹配声明广度与风险。

任务形成新提交后，提交前的证据只有在差异未改变相关输入、实现、依赖或环境时才可复用。合并、
冲突解决、生成物重建、依赖更新和配置切换通常会改变证据语义；纯文档索引提交不自动让独立代码
测试陈旧。

## 验证语义验收

宣布成功前，先明确真实场景与可观察结果：

- 前端：检查渲染后的视觉状态并实际操作相关交互；
- 持久化：保存，重启或重新加载，再回读预期状态；
- API、CLI 或自动化：使用有意义的参数与失败路径执行，再检查最终状态和**原始症状**，而不是只看响应外壳；
- 基准或代理：检查真实任务表现、异常行为和行为轨迹，而不是只看汇总分数；
- 兼容性或迁移：让**代表性消费者**、旧数据或既有产物跨越所承诺的边界运行。

单元测试、编译、lint 和静态检查只能支持这些声明，不能替代语义验收。如果证据无法触达用户相关场景，直接报告限制，不要夸大完成度。

## 收尾报告

起草收尾前，检查项目材料中定义的消费者、运行上下文与模块关系。调查没有支持之前，不要声称“没有依赖模块”或“没有剩余风险”；缺少证据不等于没有风险，也不等于没有依赖。

收尾先说证据支持的能力或产品行为，再按任务需要说明：

1. 已理解的项目架构或运维逻辑；
2. 受影响模块及边界如何关联；
3. 有意义的行为证据，包括命令、退出码/失败数和适用时的真实 UX、消费者或运维回读；
4. 证据限制、剩余阻塞、风险或下一个有意义的里程碑；
5. 提交、发布、推送等已授权且实际发生的仓库状态。

除非用户要求工程细节，或必须用标识符说明真实阻塞，否则不要以文件名、函数名、变量、工具调用或测试清单开场。这不是删除有用细节的理由：当业务、产品、运维、语义和架构证据能解释结果时，应具体说明。

## 声明之前

在内部确认三件事：可用的当前可信证据支持具体声明；重要语义验收场景已经执行；所有剩余缺口都在收尾中说清。工作者或代理报告、diff 存在、单个绿色支持信号都不能替代这个判断。

## 红旗

- 只看命令最后一行或退出码，不读失败数、跳过项和警告；
- 测试通过就声称构建、打包、迁移或真实消费者也通过；
- 代码改变后依赖旧输出，却没有判断证据新鲜度；
- 代理报告 DONE 就直接转述完成；
- 原始症状、失败路径或持久化回读从未执行；
- 因为无法验证就声称没有风险，而不是报告证据限制。
