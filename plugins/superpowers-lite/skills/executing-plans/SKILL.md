# 执行计划

行动前读取计划、稳定引用、仓库状态和进度证据。按**有意义的语义交付**推进，不按固定任务大小或仪式化待办清单机械执行。

## 建立会话任务状态

书面计划包含**两个或更多有意义的交付**、预计跨多个工具调用或对话轮次，或者上下文压缩、中断
恢复和用户可见进度会受益时，在修改实现文件前调用 Codex `update_plan`。只有一个短小、可一次
完成的交付时可以不创建任务表，但仍要遵守计划的验证、提交和恢复边界。

任务表映射语义交付，**不是微步骤**清单：不要把每条命令、测试运行、RED/GREEN 动作或 Git 操作
各拆成一个 todo。创建和维护状态时：

1. 保持**恰好一个** `in_progress`，尚未开始的交付为 `pending`；
2. 当前交付完成验证并提交成功后才能标记 `completed`，再让下一交付进入 `in_progress`；
3. 根据证据合并、拆分、重排或替换陈旧任务时，同步更新 `update_plan` 并简述原因；
4. 不为重复命令或没有语义状态变化的动作刷新任务表。

`update_plan` 是会话内进度与恢复表，不能替代持久计划、Git 提交或验证证据。当前 Codex 工具面
未提供它时，明确报告限制，并在持久计划或等价进度记录中保存同样的交付状态；不要静默跳过。

## 保持计划有效

先确认计划仍符合当前代码、约束和用户意图。证据表明安全路径已经变化时，应**调整陈旧步骤**并记录原因，不能照抄已经被代码现状淘汰的操作。

计划不是不可变施工单。主模型可以根据代码和新证据**调整任务顺序、合并或拆分任务**、
替换实现方法或补充达成目标所必需的工作。重大偏离必须记录原因和证据，使恢复者能理解
当前构造；普通实现细节变化不需要重新规划或逐项请示。

只有目标、范围、权限或硬约束发生变化才暂停；高影响阻塞和权限歧义也属于该边界。
暂停只请求解除阻塞所需的最小信息，不自行扩大权限或借机扩展范围。

开始前读取计划头中的仓库操作边界并确认**提交权限**。执行计划要求每个任务形成独立 Git
提交；如果当前没有提交权限，应在修改前一次性取得授权。不能先堆积多个任务的未提交差异，
最后再把授权问题丢给用户。

## 推进交付

对每项交付：

1. 确认依赖和当前**恢复检查点**，检查工作树中是否混有前一任务或用户的差异。
2. 完成边界清晰的修改。
3. 获取下游继续所需的聚焦证据，记录观察结果、计划偏离和遗留风险。
4. 做**失败分类**：实现缺陷应修复；环境或基础设施问题保留原始证据；计划假设失效则调整计划；
   权限或硬约束冲突必须暂停。可安全撤销的实验使用**可逆回退**，不要让失败试验污染下一任务。
5. 验证通过后立即提交，只暂存当前任务拥有的文件并检查 staged diff。提交信息说明该任务交付的
   行为或边界；**提交失败时不得进入下一任务**。
6. 提交成功后再同步 `update_plan`、持久计划或等价进度记录并标记完成，使恢复者能从提交、计划
   和证据继续。

“最终统一审查”不等于“最后统一提交”。最终审查面对的是一串可恢复、可定位的任务提交；
Critical 或 Important 集中修复也应形成单独修复提交。

只有 worktree、委派、评审或分支收尾在当前 Codex 环境和仓库策略下确实提高收益时，才使用对应工作流。不要假设必须公告、提交、推送或修改主分支。

全部交付完成后再对组合结果进行严格规格、质量、集成和回归审查。收尾时报告已交付能力、
架构关系、当前证据和剩余风险。真实的分支、集成、保留或丢弃决定必须进入仓库允许且已
授权的流程。
