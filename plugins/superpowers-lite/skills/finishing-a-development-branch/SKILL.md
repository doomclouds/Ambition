# 收尾开发分支

只在存在**真实分支**、集成或丢弃决定且有权行动时使用。先读取**当前仓库策略**和用户指令；
测试通过本身不授予合并、推送、创建 PR 或删除权限。

## 先建立真实状态

在提出选项前检查：

```console
git status --short --branch
git branch --show-current
git rev-parse --git-dir
git rev-parse --git-common-dir
git worktree list --porcelain
git log --oneline --decorate -10
```

识别普通分支、链接 worktree、宿主管理的 **detached HEAD**、上游跟踪关系和基线分支。保留并
解释任何**未提交差异**：它可能属于用户、尚未完成的任务或生成物，不能为得到干净状态而擅自
暂存、覆盖或删除。

需要根据远端状态决定 push、PR 或合并时，先判断现有 tracking ref 是否足够新；只有当前授权
允许网络操作时才 fetch。发现**远端漂移**、目标分支前进或历史分叉，应重新计算评审范围和集成
策略，不能沿用旧 merge-base 假装没有变化。

## 选择适用收尾路径

根据仓库策略、当前检出形态和用户指令提供适用选项，不固定数量：

- 保留当前分支或 detached 工作区供后续处理；
- 按仓库策略合并到目标分支，并保留要求的 merge 节点；
- 推送当前分支并创建 PR；
- 在明确展示影响和可恢复性后丢弃。

detached HEAD 不能直接提供“切回主分支并本地合并”这种假选项；需要先创建可命名引用，或使用
宿主提供的交接能力。没有 push、PR 或删除授权时，只报告下一步，不代替用户执行；禁止任何
**未经授权的推送**或强制推送。

## 集成与验证

集成前确认每个计划任务已有独立提交、工作树没有混入未归属差异，并判断现有验证是否仍与当前
HEAD 匹配。合并或 rebase 发生冲突、钩子失败、目标分支变化或验证回归都属于**集成失败**；保留
冲突与命令证据，停止清理，不把部分集成状态报告成成功。

合并成功后必须在**合并结果上重新验证**与完成声明相称的行为。PR 路径至少验证将要推送的精确
HEAD；本地分支通过并不证明合并结果或远端检查通过。

## 清理顺序

若选择合并后清理，只有合并和合并后验证成功才允许清理工作树或删除分支。

只清理当前流程创建、所有权明确且位于批准目录的工作树；Codex、宿主或外部工具管理的
工作区必须保留。删除已合并分支前，先移除工作树，再删除分支；不得在目标工作树内部
执行它自己的删除。

丢弃具有破坏性。先展示受影响工作和可恢复性，再取得明确确认。除非所有权和清理策略清楚，不得删除 Codex 或其他运行环境管理的工作区。

## 红旗

- 为得到“干净”状态而处理不属于当前任务的差异；
- 在 detached HEAD 上假设存在可安全删除的本地分支；
- 不检查目标分支或远端漂移，直接使用陈旧基线；
- 合并冲突或合并后测试失败仍继续删除分支和 worktree；
- PR 已创建就宣称完成，却没有确认推送 HEAD、检查结果或剩余风险。
