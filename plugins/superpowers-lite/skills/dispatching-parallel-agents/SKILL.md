# 并行委派代理

## 目的

并行的价值是缩短关键路径并提供彼此独立的判断。它依赖**上下文隔离**和清晰所有权，不是把
一项耦合工作切成多个同时改同一状态的代理。只有节省的时间与独立判断收益超过**协调成本**、冲突、
验证和恢复成本时才并行。

## 独立性判断

委派前按四个维度检查**独立性**：

- **知识：** 每项能否用自己的输入理解，还是必须共享正在形成的架构判断；
- **写入：** 文件、索引、数据库、缓存、生成物和分支是否有不重叠的所有权；
- **顺序：** 一项输出是否决定另一项的接口、测试夹具或验收含义；
- **运行：** 测试端口、账户、设备、部署环境和其他**外部状态**能否隔离。

只有四个维度都可安全切分，且并行后的**净收益**仍为正，才并行写入。共享文件不必绝对禁止，
但必须能按稳定区块或生成源安全分工；否则顺序执行。根因未定的一组相关失败先共同调查，不能
按失败文件名草率拆开。

## 派发契约

每个代理只获得完成自己任务所需的最小持久上下文：目标、工作目录、输入、文件或状态所有权、
不可改变的接口、禁止范围、输出位置、权限和预期证据。不要转发整段会话历史；也不要漏掉相邻
任务已经确定且会影响它的契约。

可用时通过 `spawn_agent` 启动独立任务，通过 `send_message` 补充不改变目标的信息，通过
`wait_agent` 等待结果。不要为了“用满并发”固定扇出数量。并行修改外部系统前必须明确每项
操作的授权、幂等性、冲突键和恢复方式；只读调查通常比并行写入更安全。

## 汇总与部分失败

代理返回后，主代理按依赖而不是返回顺序处理：

1. 核对报告、真实差异与原始证据，不能把代理状态当成完成证明。
2. 发现**部分失败**时，保留独立且已验证的成功结果；阻止依赖失败输出的任务，不机械撤销全部工作。
3. 检查多个结果是否对同一接口、状态或假设给出冲突结论；先解决冲突再集成。
4. 在共享接口、持久化状态、用户可见行为和真实影响边界上做一次**组合复验**。

各自绿色不等于集成通过。两个代理分别通过单元测试，但共同修改协议、迁移顺序或全局状态时，
必须验证组合后的消费者和恢复行为。

## 红旗

- 多个代理同时编辑同一生成源、迁移账本或共享环境，却没有所有权协议；
- 把互相依赖的接口实现与消费者实现并行派发，却未先固定契约；
- 一个结果失败后立即重做所有成功任务，或反过来无视失败继续集成；
- 只阅读摘要，不检查差异、测试输出和最终状态；
- 并行节省几分钟，却增加大量冲突解决和上下文重建。
