# 编写持久计划

## 目的

计划是跨多轮工作、上下文压缩和执行者交接所依赖的**持久执行契约**。更准确地说，
**计划是意图契约**：它保存后续执行者不能安全猜测的目标和边界，不把预想施工过程写成
不可改变的命令。假设后续执行者能力足够，但完全没有当前对话上下文；计划应保存：

- 用户意图和真实产品、业务或运维行为；
- 架构、模块关系、数据流和控制流；
- 精确约束、接口、取值、兼容边界与精确文件路径；
- 非例行技术细节、真实依赖和不能颠倒的构造顺序；
- 最终验收结果、关键语义验收场景和证据来源；
- 分支、worktree、提交权限和推送偏好；
- 中断后恢复所需状态。

计划不是泛泛原则，也不是把每次工具调用抄成流水账。实现步骤、实验和测试顺序默认可调整；
只有会影响目标、构造、兼容性、权限、最终验收结果或恢复的信息才固定下来。

## 固化事实

1. 读取用户请求、稳定规格、仓库指令、相关实现和测试，以及当前仓库状态。
2. 稳定要求优先引用来源；只在执行点重复不能推断错的精确值、接口、兼容规则和代码细节。
3. 说明行为由哪个模块拥有、边界如何跨越，以及为什么依赖关系决定当前构造顺序。
4. 检查提交、推送、分支和 worktree 偏好。计划执行要求每个任务完成后必须形成独立 Git 提交，
   因此在开始执行前必须一次性确认**提交权限**；没有权限时不能先执行多个任务再集中询问。
   没有明确权限不得推送。

否定边界不等于肯定设计。例如“不允许未经批准的远程存储”并未批准本地文件、内存存储或注入式存储。存储、失败策略、目录布局、所有权等选择缺少证据时，必须记录为**开放决策或阻塞**，写明解除条件，并让依赖工作保持条件化。

除非仓库或用户指定其他持久位置，把计划保存为 `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`。

## 自主调整边界

执行者可以根据当前代码和新证据调整任务顺序、合并或拆分交付、替换过时步骤，以及补充实现
目标所必需的工作。重大偏离记录原因和证据；只有目标、范围、权限或硬约束变化时才升级决定。
候选实现和示例不是逐字合规标准。

## 划分交付

按可独立验收的有意义交付划分任务。设置、实现、测试、文档和评审可以属于同一交付；只有后续任务依赖一个独立已验收行为时才拆分，不按固定分钟数切碎。

每个任务都是恢复边界：实现和相称验证完成后，只暂存该任务拥有的差异，检查 staged diff，
形成一个说明交付行为的提交。提交成功后才更新计划状态；**未提交不得标记完成**，提交失败也
不得开始下一任务。最终统一审查可以集中发现问题，但不能把所有任务推迟到最后一次提交。

当自动化测试能区分目标行为时使用红绿循环，但编译、lint 和单元测试只是支持证据。只为
下游依赖或已命名风险记录必要的连续性证据；完整规格、质量、集成和回归判断放在最终组合
结果上。计划从真实场景推导最终语义验收：

- 前端：检查渲染结果和关键交互；
- 持久化：保存、重启或重新加载后回读；
- API、CLI、自动化：使用有意义参数，覆盖失败路径并检查最终状态；
- 迁移与兼容：用代表性的既有消费者或工件验证承诺边界。

例行代码不写进计划；错误推断会改变行为、兼容性、构造顺序或验收证据时，才加入签名、模式、命令、代码片段或精确值。

## 文档头

每份持久计划以这些字段开头：

```markdown
# [功能名称] 实施计划

**目标：** [用户可见或运维结果]

**稳定引用：** [规格、决策、指令和仓库状态]

**仓库操作：** [已知提交/推送/分支/worktree 偏好，或仍需用户决定的一项内容]

**恢复状态：** [上下文压缩后先读什么、哪些交付可信、还缺哪些证据]
```

## 任务模板

按任务风险保留有用字段，不要求每项都填成长篇规格：

````markdown
### 任务 N：[有意义的交付]

**交付结果：** 此任务为最终目标提供什么可观察结果。

**硬约束与依赖：** 不能改变的接口、兼容、权限、模块所有权和已验收前置。

**候选路径：** 当前推荐的文件和方法；除明确标记为硬约束外，执行者可按证据调整。

**连续性证据：** 下游继续前必须可信的接口、状态、聚焦测试或回读；没有则写 none。

**恢复检查点：** 已完成状态、计划偏离、遗留风险、精确文件路径和下一依赖。

**提交边界：** 当前任务验证通过后应独立暂存和提交的文件范围；提交成功才算完成。
````

## 自审与交接

提交计划前确认：

1. 每项用户要求和稳定约束都映射到任务或明确的范围外决定。
2. 接口、精确值、兼容边界和依赖顺序前后一致。
3. 最终结果有真实语义验收；任务间只保留会影响下游可信度的连续性证据。
4. 新执行者无需重读原始对话即可从恢复状态继续。
5. 仓库操作偏好已记录，或剩余选择已准备好一次性询问。
6. 否定约束没有被偷换成未经批准的肯定设计。
7. 每个任务都有独立提交边界，任务之间不会依赖一大片未提交差异。

保存后给出基于复杂度、上下文、隔离、成本和长期连续性的推荐，再请用户在 SDD 与当前会话内执行之间选择。用户决定 SDD 时使用 `subagent-driven-development`；选择当前会话执行时使用 `executing-plans`，不要替用户暗选。
