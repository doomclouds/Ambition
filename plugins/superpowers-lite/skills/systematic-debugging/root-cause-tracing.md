# 根因追踪

## 何时使用

当错误出现在深层调用、队列消费者、数据库写入、生成器或外部工具中，而无效输入、状态或权限
来自更早位置时，从症状反向追踪。目标不是找到第一个能加 `if` 的地方，而是解释哪个触发点、
转换或缺失不变量让故障状态成为可能。

## 建立边界表

先写一张小型**边界表**，每行代表值或状态跨越的一处边界：

| 边界 | 预期输入 | 实际输入 | 转换/副作用 | 证据 | 所有者 |
| --- | --- | --- | --- | --- | --- |
| API → service | 已验证路径 | 空字符串 | 解析为当前目录 | 请求日志 | API |
| service → worker | 绝对路径 | 当前目录 | 创建工作区 | 调用参数 | worker |

从最深症状开始，逐步追问：这个**值或状态**从哪里来，谁最后写入或转换，谁本应验证，什么条件
允许它继续前进。调用链只是一个线索；异步消息、缓存、数据库、环境变量、文件和线程共享状态
可能绕开当前栈。

## 设置证据断点

在最能区分上下游责任的位置设置**证据断点**：

- 操作前记录输入、关联 ID、版本、权限和关键状态；
- 操作后记录输出、副作用和失败类别；
- 对共享状态记录写入者、前值、后值和时间；
- 对队列或重试记录消息身份、尝试次数、去重键和可见性窗口；
- 对环境问题回读进程真正收到的配置，而不是只看配置源文件。

日志必须避免泄露密钥或个人数据。探针应尽可能临时、聚焦且可移除；已经需要长期诊断的关键
边界可以转为正式可观测性。

## 追踪步骤

1. 固定可观察症状和失败判据。
2. 找出直接执行失败操作的代码或组件。
3. 记录它收到的实际输入与状态，而不是根据类型或调用名猜。
4. 沿生产者、转换、存储和调用链反向走一层，找到该层证据。
5. 重复，直到定位最早的错误触发、丢失验证或破坏不变量的位置。
6. 在源头构建反事实：如果修复该位置，原始症状和下游状态是否同时恢复？

找不到更上一层时不要凭空宣布源头。把当前点标成“最早已证实位置”，记录缺失证据，再用二分、
更早的日志、历史数据或最小复现继续缩小范围。

## 停止条件

根因追踪的**停止条件**不是“这里加保护后不报错”，而是同时满足：

- 能解释症状为什么只在已观察条件下发生；
- 有证据连接触发点、传播路径和最终故障；
- 修复点能恢复预期不变量，原始症状得到反事实验证；
- 仍需的下游保护覆盖独立风险，而非重复掩盖源头。

源头已知时，在源头修复。如果紧急情况下必须先遏制，明确标成“遏制措施”，记录它没有解决什么，并保持根因调查开放。边界守卫只有在处理独立故障模式时才有价值；它不能替代关于原始触发点的证据。

## 简例

空目录一路传入深层 `git init`。先记录实际 `cwd`，再沿工作区调用链和初始化状态追踪空值，找到
测试在 setup 完成前读取目录。在源头让未初始化访问明确失败，并验证预期工作区行为。拒绝危险
目录的守卫仍有价值，因为它覆盖“其他调用者也传入危险目录”的独立故障模式；但它不能替代源头修复。
