# 子代理驱动开发

## 目的

子代理的核心价值是**上下文隔离**：边界明确的实现可以在干净上下文中推进，而主代理维护长期连续性、构造顺序和整体判断。它不是每次编辑的强制包装。

## 所有权

主代理负责长期连续性、依赖和集成状态、架构沟通以及最终的语义验收责任。执行代理负责限定范围内的实现、聚焦验证和持久报告。除非需要独立核验、报告不能作为证据或同一问题反复失败，主代理不要重做执行代理的工作来“保持忙碌”。

## 选择新代理或恢复旧代理

派发前判断现有 Codex 代理是否保留有用任务知识、未决证据或局部实现状态。上下文仍有价值时使用 `followup_task` 恢复；需要独立视角、任务无关或旧上下文会形成偏见时使用 `spawn_agent` 新建。通过 `wait_agent` 等待状态，不能靠轮询日志假装协调。

## 持久交接

- 用 `scripts/task-brief PLAN_FILE N` 生成权威任务简报。
- 为代理指定与简报相邻的报告路径。报告保存项目意义、架构影响、语义证据、关注点、`commit_authority`，以及获准时的提交或未获准时的工作区差异。
- 只有独立评审有净收益时才创建评审包：已提交工作使用 `scripts/review-package BASE HEAD`；没有提交权限时使用 `scripts/review-package --worktree BASE [OUTPUT]`。
- 用 `.superpowers/sdd/progress.md` 维护进度账本，记录已接受交付、权限、提交或工作区差异、证据、关注点和下一依赖。上下文压缩后从账本和 `git log` 恢复，不靠回忆。

简报、报告、评审包和账本用于恢复信息，不是仪式。

## 委派契约

每次派发必须包含：工作目录、权威简报、不可推断的接口决定、报告路径、语义验收要求、禁止范围以及 `commit_authority`。

- `commit_authority: allowed`：任务上下文明确允许该代理提交。
- `commit_authority: absent`：不得提交，只报告差异、变更文件和证据。

不同代理或 worktree 之间必须显式说明工作区差异；不能假定它们看到相同未提交状态。模型能力应与歧义、风险和集成负载相称，不要意外继承高成本默认值。

派发前完成风险分类。只有不可逆外部操作，安全、权限、数据、跨模块接口、兼容性或共享状态
边界发生变更或存在可信的高影响风险，以及错误会污染后续任务时，才标记为高风险例外。
只读核对、沿用既有契约和低风险机械调整不自动触发。

不可逆外部操作，以及必须事前控制的安全、权限或数据风险，在执行前先评审
操作方案、授权和回滚或恢复路径；需要用户批准时先取得批准。前置门禁未通过不得派发执行，任务完成后的评审
只能补充验证，不能替代事前控制。

## 评审位置

默认不为每个任务创建规格审查或质量审查。执行代理完成**实现者自检**和聚焦验证后，
主代理检查原始证据、下游接口和已知 Critical；足以继续时直接接受交付。

中途独立评审只用于上述**高风险例外**。评审只覆盖已命名风险，不扩展成完整双审；独立且
不依赖该风险的任务仍可继续。需要事前控制的风险使用前置评审；其他风险可在证据产生后进行
聚焦验证。

所有任务完成后，对组合结果进行一次**最终统一审查**，检查用户要求、跨任务一致性、质量、
集成、回归和遗留风险。评审发现按依赖集中修复并组合复验；只有修复再次改变语义、风险或
集成置信度时才追加一次收口复审。

## 执行循环

1. 读取计划、稳定引用、仓库操作偏好和进度账本，只解除会改变构造的阻塞。
2. 选择下一项有意义交付，确认依赖、风险分类以及恢复或新建代理。
3. 高风险需要事前控制时，先完成聚焦评审、授权和恢复路径门禁。
4. 按委派契约派发，等待代理写入报告。
5. 检查状态、报告和原始证据，确认下游依赖可用且没有已知污染风险。
6. 需要事后独立证明时做聚焦评审，否则接受当前交付；更新账本并继续。

## 状态协议

- `DONE`：报告完整，进入验收或适用评审。
- `DONE_WITH_CONCERNS`：先记录关注点，再决定是否核验或跟进。
- `NEEDS_CONTEXT`：补充缺失决定或稳定引用，并在上下文仍有价值时恢复代理。
- `BLOCKED`：改变任务上下文、模型能力或拆分方式，或升级真实阻塞；无新信息时不重复派发。

## 红旗

- 把代理报告当作无需检查的真相。
- 无视有用旧上下文，机械地每任务新建代理。
- 恢复无关旧代理，污染独立判断。
- 主代理无核验理由却重复实现代理工作。
- 对普通任务逐项进行完整规格和质量双审，却漏掉真正的安全、数据、权限或跨模块风险。
- 让最终评审替代会影响后续构造的必要连续性验证和进度账本。
