# 接收代码评审

读取全部反馈后，对照代码、要求和当前行为核验。评审意见是待评估证据，不是命令，也不是社交
表演。核心顺序是**先验证再实施**。

## 建立反馈地图

记录每条反馈的**反馈来源**、位置、声称的问题、可能影响和它依赖的其他反馈。来源影响需要的
核验强度，但不决定真伪：用户给出的目标优先，外部评审仍需检查代码事实；自动化扫描可能准确
定位模式，却不了解运行契约。

把共享接口、同一根因、同一迁移或同一安全假设的事项**成组处理**。不要按评论顺序机械修改，
也不要把独立的拼写修正绑在一个未澄清的架构问题后面。

每条反馈归入一种状态：

- **已证实：** 当前代码、运行行为或要求能支持发现；
- **需要澄清：** 多种解释会改变接口、范围或风险；
- **无法验证：** 缺少环境、数据、消费者或权限，必须报告证据限制；
- **不成立：** 证据与发现相反，给出技术理由；
- **范围外或已替代：** 问题可能真实，但不属于当前目标或已由另一决定覆盖。

只有歧义会改变意图、耦合或安全时才澄清。独立且明确的事项可以在另一个承重问题解决期间继续，不把无关修复绑成单项仪式。接受发现时写清观察到的问题、选择的修改和验证方式。

如果未澄清事项会影响接口、数据状态、耦合、安全或整体意图，
未澄清前不得实施依赖它的反馈。真正独立且明确的事项仍可继续，但必须说明其不依赖当前阻塞。

建议与既有架构冲突时，检查其意图、上下游耦合、安全后果、兼容约束和真实使用证据。对“更
专业”“以后可能需要”或新增抽象的建议，先搜索**实际使用**、公开契约和已有扩展点；没有消费者
不能自动证明应删除，但也不能用假想未来为扩展范围背书。

证据表明建议会破坏契约、架构或实际场景时，应以事实反驳或请求更精确说明；不要因为评审者
身份直接照做。当前条件下无法验证时，说明需要什么输入以及哪些修改因此保持暂停，不猜测实施。

多条反馈只有彼此**独立且明确**时才能并行处理。相互影响接口、数据状态或安全假设时先协调顺序。
优先处理会导致数据损失、安全问题、破坏构建或污染后续修复的事项，再处理同根因的行为问题，
最后处理有明确价值的维护性改进。回应保持工程事实；感谢式措辞不是工作流要求。

每个独立修复组先取得对应的聚焦验证，再形成自己的 Git 提交；不要把多组已完成反馈长期留在
工作树里。只有修复改变语义、风险或集成置信度时才再次独立评审；纯机械修正保留验证证据和
无需复审的理由。最终统一复验仍检查各修复组的组合结果。
