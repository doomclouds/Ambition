# 请求代码评审

## 目的

独立评审提供实现报告无法提供的视角。默认在全部任务完成后检查**组合结果**，不要为了找
样式问题、证明流程存在或逐字核对候选步骤而制造评审。低风险机械工作不需要独立评审。

## 何时值得评审

实质性整体修改收尾前，应让评审者检查组合后的需求覆盖、语义行为、跨任务一致性、质量、
集成、回归和遗留风险，而不是只看孤立差异。

中途评审只用于不可逆外部操作，安全、数据完整性、权限、并发、兼容、共享状态或跨模块
边界发生变更或存在可信的高影响风险，以及会污染后续任务的错误，并只覆盖已命名风险。
只读核对、沿用既有契约和低风险机械调整不自动触发；不要机械地把所有改动升级成独立评审。

不可逆操作和必须事前控制的安全、权限或数据风险应在执行前审查方案、授权和恢复路径；
任务完成后的评审不能补办事前门禁。

## 输入与范围

提供允许真实判断的最小持久包：

- 要求或任务简报以及约束；
- 作为待核对证据的执行报告；
- 精确提交范围或工作区的评审包；
- 语义场景、验收证据、集成边界和已命名风险。

要求评审者判断语义行为和集成：是否解决真实问题、保持契约并与相邻模块协作。判断优先级
为用户当前意图、安全与兼容硬约束、已批准设计、真实代码契约和运行证据、计划建议步骤。
不能仅因实现偏离计划建议步骤就提出发现；必须说明实际破坏的目标、契约或风险边界。

样式和可维护性只有在威胁当前行为或未来安全修改时才构成发现，不设问题配额。

## 处理发现

依赖评审结果前，按依赖关系集中修复 Critical 和 Important，再运行一次组合复验。Minor 只有
明确影响、负责人或未来决定时才记录。评审结论与架构冲突时，用代码、要求和运行证据核对，
不盲从也不压制。

修复只有在改变语义、风险或集成置信度时才再次评审；局部机械修正保留聚焦验证和无需复审的理由。

使用 Codex 可用的 `spawn_agent` 创建只读评审代理，能力与差异风险相称；禁止固定评审扇出。整体评审模板见 [code-reviewer.md](code-reviewer.md)。
