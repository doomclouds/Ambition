# 使用任务专用操作手册

## 核心规则

技能是**任务专用操作手册**，不是每个请求都要机械展开的固定流程。用户明确点名的
技能必须读取；对于设计、排障、实施、评审或完成前的**实质性工作**，先检查可用技能，
再按任务类型选择直接匹配的手册。

加载技能是为了采用与当前任务相关的指导，而不是逐字执行全部步骤。技能明确标记为
硬门禁的**权限、安全和验证边界**必须保留；其余步骤按任务风险与规模比例适配。

## 选择与加载

1. 读取用户请求和项目规则，先识别操作是否会修改本地文件、影响外部系统或触及安全边界。
2. 对实质性工作扫描可用技能，选择最直接匹配的一个。
3. **只加载一个主要工作流技能**；当任务进入新的阶段、出现新的风险边界，或当前技能明确产出
   下一阶段的交接工件时，再加载对应技能。
4. 执行与风险相称的检查、授权确认和验证，并清楚说明未运行的验证及原因。

“一个主要工作流”限制的是同一阶段内的竞争流程，不得用来跳过**阶段转换**。设计获批后可以进入
规划；**书面计划进入执行**时必须加载 `executing-plans` 或 `subagent-driven-development`；实现
完成后再按风险进入验证、评审或分支收尾。前一技能的内容仍是下一阶段输入，不需要同时保持加载。

直接问答、窄范围查询和**低风险机械修改**可以不加载工作流技能，但仍须保留与该操作
相称的权限、安全和验证边界。例如，用户已授权的 README 五行文案修正，应先核对目标
文件和未提交改动，只修改指定文案，再用 diff 与回读验证；通常不需要完整构建。

## 常见路由

| 任务信号 | 首选技能 | 需要保留的门禁 |
| --- | --- | --- |
| 需求或架构选择尚未确定 | `brainstorming` | 先确认边界再实施 |
| 根因不明、反复失败或跨组件异常 | `systematic-debugging` | 先收集证据再修复 |
| 有可自动验证行为的新增或修复 | `test-driven-development` | 先 RED，再最小 GREEN |
| 多轮实现或需要交接 | `writing-plans` | 计划与执行检查点 |
| 已有书面计划要在当前会话执行 | `executing-plans` | 建立任务与恢复状态后实施 |
| 即将声称完成 | `verification-before-completion` | 运行相称的验证 |

这些不是所有任务必须串行执行的清单；但某个技能已经产出明确的下一阶段工件时，应遵守它声明的
交接，不能因为流程整体可适配就回到无工作流执行。

## Codex 环境

仅在需要了解 Codex 工具映射时阅读[Codex 工具说明](references/codex-tools.md)。执行书面计划并
维护多交付任务状态时应读取其中的 `update_plan` 映射。该参考只
解释工具能力，不改变用户授权、仓库规则或验证责任。

## 权限优先级

用户直接指令、仓库 `AGENTS.md` 和项目政策优先于本技能。对于提交、推送、发布、外部
消息或不可逆操作，必须依据当前授权决定；不要把工作流手册当成扩大权限的理由。
